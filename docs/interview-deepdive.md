这份文档主要整理面试官更容易深挖的问题，重点放在：

- Redis / Redis Stream
- RocketMQ 与消息队列迁移
- SQL 安全
- Java 调 Python
- 异步一致性
- 架构设计

---

## 1. Redis 相关追问

### Q1：为什么这里用 Redis Stream，而不是 RabbitMQ / Kafka / Redis List？

A：

这个项目里我选 Redis Stream，主要是考虑到当前场景是中小型项目，需要快速实现异步事件流，但又希望具备比 List 更完整的消息能力。

Redis Stream 的优势有：

- 支持消费者组
- 支持消息 ack
- 支持 pending list
- 支持失败后保留消息，便于后续重试
- 和项目里已有 Redis 依赖复用成本低

如果是更大规模、吞吐更高、链路更复杂的场景，我会考虑 Kafka 或 RabbitMQ。

---

### Q2：Redis Stream 在你这个项目里怎么用的？

A：

我这里有两个 stream：

- `stream:activity:event`
- `stream:reward:event`

参与活动成功后，后端会往 `stream:activity:event` 发送一条参与事件。

奖励请求成功后，后端会往 `stream:reward:event` 发送一条奖励事件。

然后后端内部会启动消费者监听：

- 参与事件消费者负责更新 `activity_statistics.participant_count`
- 奖励事件消费者负责更新 `reward_record.send_status`，以及 `activity_statistics.reward_count` 和 `reward_success_count`

成功后会 `ack`，失败则只记录日志，不 `ack`，这样消息还会留在 pending list，后续可以重试。

---

### Q3：为什么失败后不 ack？

A：

因为不 ack 才能保留消息状态。

如果消费失败了还去 ack，那这条消息就相当于被系统认为“已经成功处理”，但实际上数据库可能还没更新，这样会导致统计不一致。

不 ack 的好处是：

- 消息还会留在 pending list
- 后续可以人工处理
- 也可以通过定时任务自动 claim 后重试
- 更有利于保证最终一致性

---

### Q4：Redis Stream 发生重复消费怎么办？

A：

这是我在设计时特别考虑的点。

如果消息有重复投递，最危险的是统计数据被多次累加，所以我没有采用简单 `+1` 的方式，而是按“活动 + 日期”重算统计。

比如用户参与活动事件来了以后，我会重新查当天该活动真实的参与人数，再把 `participant_count` 覆盖写回。

奖励事件也是同样思路，会重算当天的：

- `reward_count`
- `reward_success_count`

这样即使重复消费，最终结果仍然是对的。

---

### Q5：Redis Stream 还有哪些可以继续优化？

A：

可以继续优化的点有：

1. 增加 pending 消息定时扫描
2. 增加 `XCLAIM` / `XAUTOCLAIM` 做消息重认领
3. 增加失败次数记录，超过阈值进入死信队列
4. 增加消费监控，比如堆积消息数、消费失败率
5. 把消费者并发和批量拉取参数做成可配置

## 1.1 RocketMQ 迁移相关追问

### Q5A：既然已经用了 Redis Stream，为什么还要迁移到 RocketMQ？

A：

Redis Stream 适合中小型项目快速落地，我用它先验证了参与事件和奖励事件的异步链路。但随着项目继续完善，我希望补充更标准的 Topic、Tag、消费组、Broker 重试和死信队列能力，所以迁移到了 RocketMQ。

迁移不是因为 Redis Stream 完全不可用，而是一次技术演进：

- Redis Stream 负责快速验证方案
- RocketMQ 作为当前默认消息队列
- Redis Stream 代码继续保留，用于技术对比和回滚验证

---

### Q5B：RocketMQ 在项目里是怎么设计的？

A：

项目使用一个 Topic：

```text
agent-task-topic
```

通过两个 Tag 区分业务任务：

```text
PARTICIPATE
REWARD
```

生产组是 `agent-task-producer-group`，消费组是
`agent-task-consumer-group`。消费者使用
`PARTICIPATE || REWARD` 过滤表达式订阅两类消息，再根据
`eventType` 分发到不同处理逻辑。

---

### Q5C：RocketMQ 消费失败后怎么处理？

A：

RocketMQ Spring 消费者不需要像 Redis Stream 一样手动 ACK。
`onMessage` 正常结束表示成功；如果业务处理抛出异常，Broker 会按照
重试策略重新投递。超过最大重试次数后，消息会进入该消费组对应的
死信队列。

生产环境中我还会继续补充：

1. 重试次数和延迟级别配置
2. 消费失败告警
3. 死信队列巡检
4. 人工或自动补偿消费

---

### Q5D：迁移后如何保证重复消费不会造成数据错误？

A：

参与任务按“活动 + 日期”从源表重新统计，再覆盖写入统计表，所以重复
投递只会得到相同结果。

奖励任务使用 `reward_record.send_status` 作为幂等状态。记录已经离开
初始状态后，重复消息不会再次执行状态流转，只会刷新派生统计。

---

### Q5E：为什么不直接删除 Redis Stream 代码？

A：

当前保留旧代码有三个目的：

1. 展示从轻量方案迁移到专业消息队列的过程
2. 便于比较 ACK、pending、Broker 重试和 DLQ 的差异
3. 在联调阶段保留回滚验证能力

Redis Stream 通过配置开关默认关闭，避免两套消费者同时处理消息。

## 2. SQL 安全相关追问

### Q6：为什么 Text-to-SQL 项目一定要做 SQL 安全控制？

A：

因为大模型生成的 SQL 不一定可靠。

如果没有安全控制，模型可能会生成：

- `DELETE`
- `UPDATE`
- `DROP`
- 多语句执行
- 查询敏感字段

这在生产环境里风险非常大。

所以 Text-to-SQL 项目不能只关注“能不能查出来”，还必须关注“会不会查出不该查的内容、会不会执行危险操作”。

---

### Q7：你具体做了哪些 SQL 安全控制？

A：

我主要做了这些限制：

1. 只允许 `SELECT`
2. 禁止 `INSERT / UPDATE / DELETE / DROP / ALTER / TRUNCATE / CREATE`
3. 禁止多语句执行
4. 禁止查询 `sys_user.password`
5. 如果 SQL 没有 `LIMIT`，自动补 `LIMIT 100`
6. LangChain 侧只暴露必要表，不把所有表都给模型

这几层组合起来，基本把模型查询限制在“只读、有限、可控”的范围里。

---

### Q8：为什么禁止查 `sys_user.password`？

A：

因为这个项目的 Agent 场景是活动运营分析，不需要查密码字段。

即使数据库里密码是加密存储，也不应该让大模型接触这个字段，因为这属于明显的敏感信息暴露风险。

所以这里不是“技术上能不能查”，而是“业务上根本不该查”。

---

### Q9：为什么还要自动补 `LIMIT 100`？

A：

主要是控制查询结果规模。

因为如果模型生成了一个不带限制的大查询：

- 会拖慢数据库查询
- 会拖慢 LLM 对结果的总结
- 会增加接口响应时间
- 还可能返回太多无用数据

所以默认补 `LIMIT 100`，相当于加一道保护。

---

## 3. Java 调 Python 相关追问

### Q10：为什么 Java 后端不直接做 Text-to-SQL，而要调 Python？

A：

主要是语言生态差异。

Java 很适合做标准业务系统，比如：

- 登录
- CRUD
- 数据落库
- 权限控制
- 缓存和消息处理

但在 AI Agent、LangChain、Prompt 编排这一块，Python 生态更成熟，开发效率也更高。

所以我采用的是：

- Java 做业务主系统
- Python 做智能查询服务

这样分工更合理，也方便后续独立升级 Agent。

---

### Q11：Java 后端是怎么调用 Python 服务的？

A：

我这里用的是 `RestTemplate`。

调用流程是：

1. Java 接收 `/agent/query`
2. 组装请求体，传 `question` 和 `user_id`
3. 调用 Python 的 `http://localhost:8000/agent/query`
4. 接收返回的：
   - `generated_sql`
   - `query_result`
   - `answer`
   - `success`
   - `error_message`
5. 把问答记录落到 `agent_qa_record`
6. 再返回给前端

调用地址放在 `application.yml`，超时时间也是可配置的。

---

### Q12：如果 Python 服务挂了怎么办？

A：

当前实现里，Java 侧会捕获调用异常并返回友好的业务错误，不会把底层堆栈直接暴露给前端。

如果进一步优化，我会做：

1. 熔断降级
2. 超时重试
3. 调用链日志
4. 失败告警
5. 健康检查和服务探活

---

### Q13：为什么要把 Python 返回的结果保存到数据库？

A：

因为这个项目最不好排查的问题，不是普通接口异常，而是模型生成了错误 SQL 或回答不稳定。

把以下内容保存下来：

- 用户问题
- 生成 SQL
- 查询结果
- 最终回答
- 成功失败状态

这样可以：

1. 复盘模型行为
2. 排查用户反馈问题
3. 优化 prompt
4. 支持后续做问答历史和审计

## 4. 异步一致性相关追问

### Q14：你这个项目里怎么理解“最终一致性”？

A：

我这里的主流程和统计流程是拆开的。

比如用户参与活动时，真正的核心数据是 `activity_user_record` 是否写成功。  
统计表 `activity_statistics` 不是主交易数据，而是汇总数据，所以它可以稍后异步更新。

因此这里不是强一致，而是最终一致：

- 主流程先成功
- 异步消费者再刷新统计
- 如果消费失败，消息保留待重试
- 最终让统计数据和业务明细保持一致

---

### Q15：为什么奖励记录要先写 `INIT` 状态？

A：

因为奖励发放本身是异步动作。

如果主流程里直接写成成功，就会让数据库状态和实际消费过程不一致。

所以更合理的方式是：

1. 主流程先写 `INIT`
2. 发消息到 Redis Stream
3. 消费者真正处理完成后改成 `SUCCESS`
4. 如果失败，可以保留失败原因或者留待后续重试

这样状态流转更清晰，也更贴近真实业务。

---

### Q16：如果消息发出去了，但是数据库事务回滚了怎么办？

A：

这是典型的“本地事务和消息发送一致性”问题。

当前这个项目是练手项目，逻辑上已经做到“写库后再发消息”，但如果要更严谨，我会考虑：

1. 本地消息表
2. 事务消息
3. Outbox Pattern

因为严格来说，“数据库写入成功”和“消息一定发送成功”之间还存在边界问题。

这个点如果在面试里能主动说出来，会比较加分。

## 5. 架构和设计相关追问

### Q17：为什么要单独设计 `activity_statistics` 表？

A：

因为运营查询很多都是统计类、聚合类查询。

如果每次都直接扫明细表：

- SQL 会更复杂
- 性能更差
- 高并发时对数据库压力更大

所以把每日统计单独汇总出来，是典型的业务统计表设计。

---

### Q18：为什么要分 controller / service / mapper / dto / vo？

A：

这样分层主要是为了职责清晰。

- `controller`：接收请求、参数校验、返回结果
- `service`：写业务逻辑
- `mapper`：和数据库交互
- `dto`：接收入参
- `vo`：输出结果
- `entity`：数据库实体

这样后续维护、测试、扩展都更方便。

---

### Q19：这个项目里你最想强调的工程思维是什么？

A：

我最想强调三个点：

1. 安全意识  
   Text-to-SQL 一定要做 SQL 安全控制。

2. 一致性意识  
   异步统计更新要考虑消息重试和重复消费问题。

3. 可排障意识  
   生成 SQL 和问答结果都要落库，方便问题排查和后续优化。
